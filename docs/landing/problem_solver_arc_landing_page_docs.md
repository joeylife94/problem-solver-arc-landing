# ProblemSolverArc Landing Page Production Document V1.0

## 0. Purpose of This Document
This document defines the production standard for the ProblemSolverArc freelance landing page.
The goal is not to create a visually impressive introduction page, but to build a business-card-style landing page that can be taken to market immediately.

This document is intended to serve as the reference for:
* Landing page implementation
* Copywriting
* Service package definition
* Scope control
* Next-session handoff

---

## 1. Project Overview
* **Project Name:** ProblemSolverArc
* **Project Type:** A freelance business-card-style landing page
* **Primary Role:** An entry page for portfolio presentation and service sales
* **Language Strategy:** Build in Korean first; Keep the structure expandable for future multilingual support
* **Current Stage:** V1.0 documentation phase
* **Next Stage:** Actual landing page v0.1 implementation

---

## 2. Role of the Landing Page
This landing page should communicate the following four things in a single screen flow:
1. Who I am
2. What kind of problems I solve
3. What services I offer
4. How to contact me

This is not just a self-introduction page. Its core purpose is to communicate the following message:
> "This person can define scope clearly and build small operational systems, internal tools, and backend-centered MVPs in a structured way."

---

## 3. Positioning
* **Main Positioning Statement:** I build internal tools and backend systems with clear scope and validation.
* **Sub Positioning Statement:** For small teams and real operations — from messy workflows to structured, workable software.
* **Core Positioning Keywords:** clear scope, validation, internal tools, backend systems, real operations, structured software.

---

## 4. Target Customers
The current landing page primarily targets the following customer groups:

**Primary Targets**
* Small teams
* Early-stage product teams
* Small businesses with real operational workflows
* Customers who need internal tools, admin dashboards, or backend MVPs

**Examples of Good-Fit Customers**
* Teams that want to replace manual or spreadsheet-based operations with software
* Businesses that need admin functionality to organize internal workflows
* Teams that need to reduce scope and build an MVP quickly
* Customers who need a small working system, not just a demo

**Examples of Poor-Fit Customers**
* Full-scale large platform development requests
* Clients who only want a high-end design-focused branding website
* Clients who want app, web, branding, and marketing as one bundled package
* Clients who expect unlimited feature expansion or long-term maintenance by default

---

## 5. Core Message
The page should communicate the following messages in this order:

1. **Identity:** I build internal tools and backend systems for small teams.
2. **Working Style:** I define scope first, build structurally, and deliver results that can be validated.
3. **Services:** I support projects in stages, from lightweight landing pages to internal tools and backend-centered MVPs.
4. **Proof of Credibility:** I present an operations MVP centered around audit/logging as a featured example.
5. **Call to Action:** Clients can send me their project idea or operational workflow by email.

---

## 6. Service Structure
The services will follow a three-tier structure.
* Prices should be shown as *Starting at*.
* The initial pricing currency will be KRW.

### Service 1. Validation Landing
* **Pricing:** Starting at 590,000 KRW
* **Description:** A lightweight landing page built for idea validation, core message delivery, and inquiry generation.
* **Good Fit For:**
  * Validating an idea quickly
  * Needing a first page to introduce a business
  * Needing a basic page to receive inquiries
* **Includes:** One landing page, responsive web design, CTA / inquiry-focused structure, deployable basic build, source code delivery.
* **Excludes:** Admin pages, payment functionality, advanced custom branding design, complex backend workflows.

### Service 2. Internal Tool / Admin MVP
* **Pricing:** Starting at 1,290,000 KRW
* **Description:** A small internal tool or admin MVP that helps organize real operational workflows.
* **Good Fit For:**
  * Replacing manual or spreadsheet-based tasks with a tool
  * Building an admin interface for internal operations
  * Creating a small operational system that can be used in practice
* **Includes:** Admin or internal operations interface, basic CRUD flows, API + database integration, practical backend structure, source code delivery.
* **Excludes:** Large-scale permission systems, complex external system integrations, advanced commerce or settlement logic, large multi-service architectures.

### Service 3. Backend MVP Sprint
* **Pricing:** Starting at 2,490,000 KRW
* **Description:** A short-sprint build for a backend-centered MVP where the core workflow actually works in production-oriented form, not just as a demo.
* **Good Fit For:**
  * Small teams that need to build an MVP for real use
  * Products where backend structure matters most
  * Services with operational workflows that need to start small
* **Includes:** Core feature scope definition, API / DB structure design, core backend implementation, partial admin functionality if needed, deployable deliverable, source code delivery.
* **Excludes:** Full product development for an entire startup service, unlimited feature expansion, app + web + branding full-package work, large platform-scale development.

## 7. Pricing Policy
* **Basic Principles:** Prices are displayed in *Starting at* format. The final quote may vary depending on scope, complexity, and region. Starting prices apply only to clearly scoped projects.
* **Current Base Prices:**
  * Validation Landing: from 590,000 KRW
  * Internal Tool / Admin MVP: from 1,290,000 KRW
  * Backend MVP Sprint: from 2,490,000 KRW
* **Draft Pricing Copy:** "Final quote depends on scope, complexity, and region. Starting prices are for clearly scoped projects only."
* **Operating Principles:** Do not implement automatic currency conversion in the initial version. Multilingual expansion may come later, but currency policy should be handled in a separate phase.

---

## 8. Contact Method
* **Current Decision:** The initial contact channel will be **email only**. 
* **Why:** Easiest to manage, leaves a structured written record, best for organizing scope, quote, and requirements. (Expansion to KakaoTalk or Instagram will be postponed to a later stage.)
* **CTA Copy:** "Send your workflow or MVP idea. Email me with your project type, goal, needed features, timeline, and budget range."
* **Helpful Information to Ask For:** Project type, goal, required features, desired timeline, budget range.
* **Display Rule:** The Contact section should display a direct email address below the CTA copy.

## 11. Working Style Messages
The landing page should clearly communicate the following four principles:

1. **Clear scope before build:** Cut scope first, then build.
2. **Backend-first thinking:** Prioritize workflow, structure, and operational logic before UI.
3. **Validation-minded delivery:** Do not just build and stop; emphasize outcomes that can be tested and validated.
4. **Built for real operations:** Aim for structures that fit real operational workflows, not demo-only products.

---

## 12. What Will Not Be Done in This Version
The following items will not be included in V1 or v0.1 implementation:
* SeoulGyeol landing page work
* Review and inclusion of other GitHub projects
* Multilingual functionality
* Automatic currency conversion
* Refined animations
* Excessive branding design work
* Full case study body writing
* Additional service package expansion
* Excessive AI-marketing-style messaging

**Core Principle:** This version is not a polished full-brand website. It is a business-card-style landing page that can be taken to market immediately.

## 13. v0.1 Implementation Goal
The actual landing page v0.1 will be considered sufficient if it satisfies the following conditions:

**Success Criteria:**
* Identity, service, trust, and contact method are visible in a single-screen flow.
* The differences between Service 1, 2, and 3 are easy to understand.
* The featured project fits naturally into the page.
* The email CTA does not feel awkward.
* The copy is not overly abstract.
* Message clarity matters more than visual design.
